# N. 대시보드 실시간 상태

**범위**: 프로젝트 리스트 상단 5개 카드, 구독/구독해제 라이프사이클.
**선행 도메인**: A.

---

## N1. 5개 통계 카드 `[CORE] [ASYNC]`

### N-01-01: 초기 렌더 값 검증
**절차**: `/projects` 진입.
**기대 결과**:
- 카드: 프로젝트 수 / 세션 수 / 활성 세션 / 큐 실행 / 터미널
- 각 수치가 실제 서버 상태와 일치 (e.g. 세션 리스트 합과 동일)

### N-01-02: 실시간 증감

> **SKIP 금지.** Claude API 호출 1회(짧은 프롬프트)로 충분히 검증 가능하다. API 비용이 부담되어도 `"Reply with: OK"` 같은 1~2 토큰 응답을 유도하면 $0.0001 수준이므로 이 시나리오는 실제로 실행해야 한다. "실시간 반영" 확인은 대시보드의 핵심 가치이며 스킵 시 회귀 감지 불가.

**절차**:
1. 탭 A: `/projects` 대시보드 가시 — 현재 "세션 수" 값 기록 (`N_sessions_before`)
2. 탭 B: `browser_tabs(action="new")` → 기존 비-BMad 테스트 프로젝트 → "새 세션 시작"
3. 탭 B 채팅 입력에 `"Reply with exactly: OK"` 전송 — Claude 응답까지 대기(보통 1~3초)
4. 탭 A 로 `browser_tabs(action="select", index=0)` → 1.5초 이내 대시보드 갱신 관찰
5. "세션 수" 값이 +1 되었는지 확인 (300ms 디바운스로 즉시가 아닐 수 있음)
6. 스트리밍 중간 시점의 "활성 세션" 증가 여부는 타이밍상 놓칠 수 있으니 **부가 기대값**으로 기록

**기대 결과**:
- "세션 수" = `N_sessions_before + 1` (300ms 디바운스 이후)
- (부가) 스트리밍 중 "활성 세션" +1, 스트리밍 종료 시 -1 — 관찰되면 기록, 놓쳐도 FAIL 아님 (짧은 응답 특성)

**엣지케이스**:
- E1. 빈 세션 생성만 하고 메시지 미전송: 현재 구현은 빈 세션을 "세션 수" 카운트에 포함시키지 않을 수 있음 (B-01-02 관찰). 반드시 메시지 1회 전송 후 관찰해야 함.

## N2. 구독 / 구독해제 라이프사이클 `[EDGE]`

### N-02-01: 다른 페이지로 이동 시 구독 해제
**절차**:
1. 대시보드 진입 (`dashboard:subscribe` 발송 확인)
2. 프로젝트 페이지로 이동 (`dashboard:unsubscribe` 확인)

**기대 결과**: 불필요 이벤트 스트림 중지, 서버 리소스 정리.

### N-02-02: 다중 브라우저 동시 구독
**절차**: 동일 계정 두 브라우저에서 동시에 대시보드.
**기대 결과**: 둘 다 동일 상태 수신, 서버 중복 브로드캐스트 과다 없음.

### N-02-03: 장시간 유지
**기대 결과**: WebSocket ping/pong으로 연결 유지, 재연결 시 자동 재구독.
